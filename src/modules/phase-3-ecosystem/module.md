# Phase 3 — AI Coding Ecosystem

Fase ketiga ini mengenalkan ekosistem alat dan platform AI untuk coding — dari Claude Code hingga Token. Pahamilah lanskap alat yang ada dan bagaimana semuanya terhubung.

---

## Phase 3 — AI Coding Ecosystem

### Claude Code

<Intuition>
Seperti memiliki programmer junior yang terampil duduk di sebelahmu, yang bisa menulis, mengedit, dan menjalankan kode langsung, berdasarkan instruksimu.
</Intuition>

<Theory>
Alat coding AI buatan Anthropic yang bisa menulis, mengedit, dan menjalankan kode langsung di dalam proyekmu, mengikuti instruksimu langkah demi langkah. Tidak seperti antarmuka chat di mana kamu copy-paste kode manual, Claude Code beroperasi langsung pada file-filemu.
</Theory>

<Example>
Seorang developer meminta Claude Code untuk membangun fitur baru, dan alat itu menulis file yang sebenarnya, mengujinya, dan memperbaiki error — sebagian besar otomatis.
</Example>

<Conclusion>
Alat seperti ini secara dramatis mempercepat coding, tapi masih butuh manusia yang memahami business logic dan architecture untuk membimbingnya dengan baik.
</Conclusion>

**🦊 Opportunist Lens:** Jika kamu menggabungkan fundamental yang kuat dengan alat seperti ini, kamu bisa membangun jauh lebih cepat daripada orang yang (a) tidak menggunakan alat AI, atau (b) menggunakan alat AI tanpa memahami kodenya.

---

### OpenCode

<Intuition>
Ide yang mirip dengan Claude Code — pikirkan sebagai merek berbeda dari "asisten coding AI yang bisa bertindak di dalam proyekmu," tapi open-source (kodenya publik dan gratis untuk diperiksa).
</Intuition>

<Theory>
Alat asisten coding AI open-source, mirip tujuan dengan Claude Code — membantu menulis dan mengelola kode secara langsung. Menjadi open-source berarti kode di balik alat tersedia untuk publik.
</Theory>

<Example>
Disebutkan dalam diskusi sebagai salah satu alat yang membuat teman seseorang menjadi "malas" — terlalu mengandalkannya daripada belajar kode secara manual.
</Example>

<Conclusion>
Pasar alat coding AI berubah cepat — alat baru sering muncul, dan alat terbaik hari ini mungkin bukan yang terbaik tahun depan.
</Conclusion>

**🦊 Opportunist Lens:** Jangan menikah dengan satu alat. Pelajari skill yang mendasarinya (cara bekerja dengan asisten coding AI secara umum), sehingga kamu bisa berganti alat dengan mudah saat yang lebih baik muncul.

---

### Hermes

<Intuition>
Disebutkan dalam diskusi mirip dengan "OpenClaw" (nama alat AI lain) — pikirkan sebagai pemain lain di dunia sistem asisten AI pribadi yang sedang berkembang.
</Intuition>

<Theory>
Alat/platform AI yang disebutkan dalam diskusi, digunakan untuk tugas-tugas seperti asisten pribadi (kategori yang mirip dengan Personal Assistant AI di Phase 1).
</Theory>

<Example>
"Hermes as well, bro" — direkomendasikan dalam diskusi bersama alat pembangun asisten lainnya.
</Example>

<Conclusion>
Nama alat spesifik di bidang ini datang dan pergi dengan cepat. Sumber pengetahuan seperti file ini mungkin menjadi usang — selalu periksa pembaruan.
</Conclusion>

**🦊 Opportunist Lens:** Mengetahui nama alat yang kurang dikenal lebih awal (sebelum menjadi mainstream) bisa menjadi keuntungan kecil, tapi cepat hilang — keuntungan yang lebih besar selalu workflow-mu, bukan nama alatnya.

### OpenRouter

<Intuition>
Seperti satu aplikasi yang memungkinkanmu memesan makanan dari banyak restoran berbeda, daripada mengunduh aplikasi terpisah untuk setiap restoran.
</Intuition>

<Theory>
Platform yang memungkinkanmu mengakses banyak model AI berbeda (dari perusahaan berbeda) melalui satu koneksi, daripada mendaftar secara terpisah untuk masing-masing. Satu API key, satu dashboard, satu sistem billing.
</Theory>

<Example>
Seseorang dalam diskusi mengatakan mereka menggunakan OpenRouter untuk mengakses model seperti DeepSeek dan Gemini untuk gambar, terutama berguna untuk opsi free-tier.
</Example>

<Conclusion>
Platform seperti ini populer di kalangan orang yang ingin mencampur dan mencocokkan model AI berdasarkan biaya dan jenis tugas.
</Conclusion>

**🦊 Opportunist Lens:** Menggunakan banyak model secara cerdas (murah untuk tugas sederhana, kuat untuk tugas sulit) menghemat uang — kebanyakan orang tidak repot mengoptimalkan ini.

### MCP (Model Context Protocol)

<Intuition>
Bayangkan colokan listrik universal yang bekerja di negara mana pun, daripada perlu adaptor berbeda untuk setiap stopkontak. MCP adalah "colokan universal" itu — tapi untuk menghubungkan model AI ke alat dan sumber data yang berbeda.
</Intuition>

<Theory>
Standar (seperangkat aturan yang disepakati) yang memungkinkan model AI terhubung dengan mudah ke alat, aplikasi, dan sumber data luar — daripada setiap koneksi dibuat khusus.
</Theory>

<Example>
Asisten coding AI yang menggunakan MCP bisa langsung membaca file proyekmu, atau terhubung ke aplikasi task-tracking-mu, menggunakan metode standar.
</Example>

<Conclusion>
Standar semacam ini membuat seluruh ekosistem alat AI bekerja bersama lebih lancar, itulah mengapa ini menjadi populer dengan cepat.
</Conclusion>

**🦊 Opportunist Lens:** Memahami cara menghubungkan AI ke alat-alatmu sendiri (melalui hal-hal seperti MCP) memungkinkanmu membangun workflow kustom yang tidak pernah dicoba oleh kebanyakan pengguna biasa.

### Agent (AI Agent)

<Intuition>
Chat AI sederhana seperti orang yang hanya menjawab pertanyaan. Seorang "agent" seperti orang yang benar-benar bisa melakukan sesuatu untukmu — mencari, mengklik, menulis, memperbaiki, dan melapor kembali — dengan sedikit bimbingan langkah demi langkah.
</Intuition>

<Theory>
Sistem AI yang bisa mengambil banyak tindakan sendiri (tidak hanya menjawab satu pertanyaan) untuk menyelesaikan tugas, sering menggunakan alat-alat. Perbedaan antara chatbot dan agent adalah otonomi.
</Theory>

<Example>
Seorang agent mungkin diberi tahu "perbaiki bug ini," dan dia akan menemukan bug, menulis perbaikan, menguji perbaikan, dan melapor kembali — tanpa kamu memberi tahu setiap langkahnya.
</Example>

<Conclusion>
Agent itu kuat tapi masih membuat kesalahan. Praktik yang baik: biarkan agent bekerja, tapi selalu review hasil yang penting.
</Conclusion>

**🦊 Opportunist Lens:** Belajar membimbing agent dengan baik (tidak hanya chat dengan mereka) adalah skill baru yang kebanyakan orang belum praktekkan.

### Context Engineering

<Intuition>
Jika Prompt Engineering adalah "bagaimana kamu mengajukan pertanyaan," Context Engineering adalah "bagaimana kamu menyiapkan semua informasi latar belakang" sebelum kamu bertanya.
</Intuition>

<Theory>
Skill mengorganisir dan menyiapkan informasi latar belakang yang tepat (file, riwayat, instruksi) sehingga alat AI memahami situasi penuh, bukan hanya satu pesan.
</Theory>

<Example>
Daripada bertanya pada AI satu pertanyaan acak, kamu pertama-tama memberikan file proyekmu, keputusan masa lalu, dan tujuan — itu adalah context engineering.
</Example>

<Conclusion>
Semakin kompleks tugas, semakin penting context engineering daripada kata-kata prompt saja. Prompt sempurna dengan konteks buruk tetap memberikan jawaban dangkal.
</Conclusion>

**🦊 Opportunist Lens:** Kebanyakan pengguna biasa hanya melakukan prompting dasar. Menguasai context engineering menempatkanmu di depan, terutama untuk proyek panjang dan kompleks.

### Model 3B / Model 7B

<Intuition>
Pikirkan seperti ukuran mesin di mobil — mesin kecil (efisien, lebih ringan) versus mesin lebih besar (lebih bertenaga, tapi butuh lebih banyak sumber daya).
</Intuition>

<Theory>
"3B" dan "7B" berarti "3 miliar" dan "7 miliar" parameter (pengaturan internal yang membuat model AI "berpikir"). Angka lebih besar biasanya berarti lebih kuat, tapi juga lebih berat (butuh komputer yang lebih kuat).
</Theory>

<Example>
"Untuk beberapa tugas ringan, saya menggunakan model 3B atau 7B, di-host di DeskMini Ryzen (komputer pribadi kecil) saya."
</Example>

<Conclusion>
Model lokal kecil menjadi populer untuk privasi, penghematan biaya, dan kecepatan pada tugas sederhana.
</Conclusion>

**🦊 Opportunist Lens:** Tahu kapan menggunakan model kecil (murah, cepat) versus model besar (kuat, mahal) menghemat uang dan waktu nyata — skill yang diabaikan kebanyakan pemula.

### Local Model

<Intuition>
Seperti memasak di rumah daripada memesan delivery — butuh sedikit persiapan, tapi pribadi, dan kamu tidak tergantung pada dapur orang lain (server).
</Intuition>

<Theory>
Model AI yang berjalan langsung di komputermu sendiri, bukan di server cloud perusahaan. Ini berarti semuanya tetap lokal — data tidak pernah meninggalkan mesinmu.
</Theory>

<Example>
Menjalankan model AI kecil di komputer mini pribadi (seperti "DeskMini Ryzen") untuk tugas dasar sehari-hari.
</Example>

<Conclusion>
Model lokal meningkat dengan cepat, dan lebih banyak penghobi sekarang menjalankan model AI yang berguna di rumah.
</Conclusion>

**🦊 Opportunist Lens:** Menjalankan model lokalmu sendiri memberimu lebih banyak kontrol dan privasi — berguna jika kamu membangun sesuatu yang tidak ingin dibagikan dengan perusahaan cloud.

---

### API

<Intuition>
Seperti menu restoran — kamu tidak perlu tahu bagaimana dapur memasak makanan; kamu hanya memesan dari menu, dan dapur (sistem) menangani sisanya.
</Intuition>

<Theory>
"API" adalah singkatan dari **Application Programming Interface** — cara bagi dua sistem software yang berbeda untuk "berbicara" satu sama lain, mengikuti aturan yang jelas.
</Theory>

<Example>
Ketika developer menghubungkan aplikasi mereka ke model AI seperti Gemini atau DeepSeek, mereka biasanya melakukannya "melalui API" — mengirim request dan menerima response.
</Example>

<Conclusion>
Hampir setiap produk AI yang kamu gunakan dibangun di atas API — bahkan jika kamu tidak pernah melihatnya secara langsung sebagai pengguna biasa.
</Conclusion>

**🦊 Opportunist Lens:** Memahami penggunaan API dasar memungkinkanmu membangun alat kustom sendiri, bukan hanya menggunakan aplikasi buatan orang lain.

---

### Token

<Intuition>
Pikirkan token seperti potongan puzzle teks. AI tidak membaca seluruh kata seperti manusia; ia memecah teks menjadi potongan kecil (token) dan memprosesnya.
</Intuition>

<Theory>
Unit kecil teks (kadang seluruh kata, kadang hanya bagian dari kata) yang dibaca dan dihasilkan oleh model AI. Biaya penggunaan AI sering diukur "per token."
</Theory>

<Example>
Dokumen panjang yang kamu unggah ke alat AI menggunakan banyak token; pertanyaan pendek menggunakan sangat sedikit.
</Example>

<Conclusion>
Memahami token membantu kamu menggunakan alat AI lebih efisien dan mengontrol biaya, terutama saat menggunakan API berbayar.
</Conclusion>

**🦊 Opportunist Lens:** Mengelola konteks dan token dengan cerdas berarti kamu mendapat jawaban yang lebih baik dan lebih konsisten daripada seseorang yang hanya membuang semuanya ke chat secara acak.
